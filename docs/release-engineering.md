# Release engineering

Two audiences for this document:

- **Downloaders** — verify that the SD image you just downloaded was
  built and signed by the PrepperPi maintainers, not a tampered copy
  served from a hijacked mirror. Jump to [Verifying a downloaded
  image](#verifying-a-downloaded-image).
- **Maintainers** — cut a new release, rotate the signing key, or
  triage a release-pipeline failure. Jump to [Maintainer
  reference](#maintainer-reference).

## Trust model in one paragraph

Every published release ships a compressed SD image (`.zip`), a
`SHA256SUMS` file with the image's hash, and detached GPG signatures
(`.asc`) over both the image and the checksum file. The signing key
lives only in two places: the maintainer's GPG keyring (where it was
generated) and the `GPG_PRIVATE_KEY` GitHub Actions secret on the
`jmarler/prepperpi` repo. The matching public key is committed to the
repo at [`release/prepperpi-public.asc`](../release/prepperpi-public.asc)
and uploaded as a release artifact for convenience. Anyone can verify
end-to-end with `gpg --verify` + `sha256sum -c`, no PrepperPi tooling
required.

The signing key has no passphrase. GitHub's secret encryption is the
trust boundary; a passphrase would either be a second GitHub secret
(no extra security) or a manual prompt (defeats automation). The
practical attack surface is "GitHub-side compromise of the prepperpi
repo," which would also let an attacker push poisoned source — so
adding a passphrase doesn't move the needle.

## Verifying a downloaded image

Every release page has six artifacts. From the GitHub Release at
`https://github.com/jmarler/prepperpi/releases/latest`, download:

```
prepperpi-<version>.zip               # the SD image
prepperpi-<version>.zip.asc           # signature over the .zip
SHA256SUMS                            # one-line sha256 of the .zip
SHA256SUMS.asc                        # signature over SHA256SUMS
prepperpi-public.asc                  # the public key
```

(You can skip the `.rpi-imager.json` for verification; that's used by
Raspberry Pi Imager's `--repo` mode.)

Then run the standard two-step verify:

```sh
# Import the public key (one-time per machine).
gpg --import prepperpi-public.asc

# 1. The signatures are valid and made by the PrepperPi key.
gpg --verify SHA256SUMS.asc SHA256SUMS
gpg --verify prepperpi-<version>.zip.asc prepperpi-<version>.zip

# 2. The image's bytes match what was signed.
sha256sum -c SHA256SUMS
```

Both `gpg --verify` calls should print `Good signature from "PrepperPi
Releases <release@prepperpi.org>"` and the fingerprint
`4E4C 11F5 687A DBDF 34F5  6E0F 5D5E 997E 4D04 5BFC`. The
`sha256sum -c` should print `prepperpi-<version>.zip: OK`.

If anything fails, **don't flash the image**. Re-download (a CDN edge
might be stale or corrupted), and if the failure repeats, open an
issue at <https://github.com/jmarler/prepperpi/issues>.

A note about the "WARNING: This key is not certified with a trusted
signature" line you'll see from gpg: that's normal for a project
key you imported manually. It means you haven't told gpg you trust
this key — which is fine. The verification result you care about is
whether the signature is valid; trust-level annotations are about
the OpenPGP web-of-trust, which PrepperPi doesn't participate in.

## Maintainer reference

### Cutting a release

The release pipeline triggers automatically when you push a tag
matching `v*`:

```sh
git switch main && git pull
git tag v1.2.3
git push origin v1.2.3
```

The `Release` workflow takes ~15-25 minutes on the GitHub-hosted
`ubuntu-24.04-arm` runner. When it finishes, the new release is at
<https://github.com/jmarler/prepperpi/releases>. Tags ending in
`-rc.<N>` or `-beta.<N>` (for example `v0.99.0-rc.1`) are
auto-detected and marked as **prerelease**.

A failed publish can be re-run via `workflow_dispatch` against the
same tag — the workflow uses `gh release upload --clobber` if the
release already exists, so you can iterate without re-tagging.

### Release notes

Notes are auto-generated by `gh release create --generate-notes`,
which reads [`.github/release.yml`](../.github/release.yml) to group
merged PRs into Features / Bug fixes / Documentation / Maintenance /
Other changes.

For the grouping to work, every PR needs at least one of the recognized
labels: `feat`, `enhancement`, `fix`, `bug`, `docs`, `chore`,
`dependencies`. Unlabeled PRs land in "Other changes". You can label
retroactively before tagging — the notes are generated at tag-push
time, not at PR-merge time.

### One-time signing key setup

This was done on **2026-04-29** with the project's first signing key.
Re-run only if rotating (see below).

```sh
# 1. Generate. ed25519, no expiry, signing-only, no passphrase.
gpg --quick-generate-key 'PrepperPi Releases <release@prepperpi.org>' \
    ed25519 sign never

# 2. Confirm the key is the only secret key with that uid.
gpg --list-secret-keys --keyid-format=long

# 3. Commit the public key (it's already at release/prepperpi-public.asc;
#    overwrite when rotating).
gpg --armor --export <FINGERPRINT> > release/prepperpi-public.asc

# 4. Upload the private key to the repo as the GPG_PRIVATE_KEY secret.
gh secret set GPG_PRIVATE_KEY --repo jmarler/prepperpi \
    < <(gpg --armor --export-secret-keys <FINGERPRINT>)
```

Local `gpg` keeps the private key as a backup. **Don't delete the
local copy** — losing both the GitHub secret and the local key means
no future release can sign with this identity, and downloaders with
the old public key won't trust signatures from a new one.

### Rotating the signing key

The current key has no expiration, so rotation is only triggered by
suspected compromise or when handing the project to a new maintainer.

1. Generate a new key (same `--quick-generate-key` invocation,
   different output fingerprint).
2. Update the `GPG_PRIVATE_KEY` secret to the new key's private export.
3. Replace `release/prepperpi-public.asc` with the new public key.
4. Commit the new public key with a release note flagging the
   rotation; downloaders need to re-import.
5. Optionally publish a revocation certificate for the old key
   (gpg generates one in `~/.gnupg/openpgp-revocs.d/` at
   `--quick-generate-key` time).

### Bumping pinned pi-gen

The build pins pi-gen to a specific commit SHA in
[`images/build.sh`](../images/build.sh) (`PI_GEN_DEFAULT_REF`). Bump
deliberately when you want to pick up upstream pi-gen fixes. Any
release that includes a pi-gen bump should call out what changed in
the release body — pi-gen affects what packages land in the base
image and what bootloader bits are in the boot partition.

To preview before tagging: trigger `Build SD image`
(workflow_dispatch on `build-image.yml`) with `pi_gen_ref` set to the
new SHA, confirm the build is green, then update
`PI_GEN_DEFAULT_REF` and ship.

### Checking the in-image version stamp

Every built image carries `/etc/prepperpi/image.version`:

```
image_version=v1.2.3
git_commit=abcdef0123456789...
pigen_rev=4ad56cc850fa60ad...
built_at=2026-04-29T10:11:12Z
```

This is also surfaced in the admin console's home-page footer, so a
running Pi can be matched back to a specific tag without SSH access.
